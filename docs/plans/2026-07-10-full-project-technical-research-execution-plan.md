# 全项目技术调研执行计划

## 执行等级

内部等级为 L：任务覆盖面大，但输出目标统一，采用单执行通道分阶段完成，避免多个执行者同时写文档造成事实口径不一致。

## 阶段

1. 检查分支、工作区、仓库结构和约束文件。
2. 统计后端、前端、资源、配置和部署文件，识别关键技术栈。
3. 阅读控制器、服务、实体、WebSocket、RAG、备份、前端路由和 API 层。
4. 编写文档生成器，以源文件为事实输入生成分章节逐文件索引。
5. 补充架构与关键业务链路的人工整理章节。
6. 静态核验行数、章节链接、覆盖率、超大文件和 Git 差异。
7. 只暂存本任务新增文件，创建非签名提交并推送当前分支。

## 所有权边界

- 可写：`docs/technical-reference/`、本需求文件、本计划文件、对应 runtime 凭据和文档生成脚本。
- 不可写：现有业务源码、前端源码、数据库 schema、部署脚本和用户已修改的 `README.md`。

## 核验命令

- PowerShell `Get-Content -Encoding UTF8`：验证文档可读。
- PowerShell `Measure-Object -Line`：统计文档集行数。
- `rg`：核对控制器、映射、实体、路由和 API 路径覆盖。
- `git diff --check`：检查新增文本空白错误。
- `git status --short` 与 `git diff --cached --name-only`：确认提交边界。

这些命令均为静态文本检查，不属于构建或测试。

## 回滚规则

- 生成失败时仅删除本次新增的技术文档输出，不触碰已有文件。
- 若发现用户文件冲突，停止覆盖并保留用户版本。
- 推送失败时保留本地提交并明确报告失败原因。

## 清理要求

- 不保留临时扫描文件。
- 文档生成器作为可重复维护工具保留在 `scripts/documentation/`。
- 输出 cleanup receipt，记录临时文件、工作区和交付核验结果。
